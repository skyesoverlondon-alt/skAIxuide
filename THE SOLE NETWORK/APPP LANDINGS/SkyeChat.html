Platform Overview

SkyeChat is a multi-tenant chat SaaS built as Netlify Static + Netlify Functions with Neon Postgres persistence, real-time fanout via Ably, auth and invites via Resend, and monetization via Stripe.

At its core, the platform powers two distinct access surfaces: a first-party admin and agent console served from a clean JavaScript UI with HttpOnly cookie sessions, and a third-party embeddable widget that drops into any external site via a simple iframe and JavaScript include. Both surfaces share the same backend API, the same rate limiting infrastructure, and the same immutable audit trail.

What makes SkyeChat different from the field of "chat API" products isn't what it connects to — it's what it enforces. Every message write goes through messages_send, which triggers DLP enforcement, moderation hooks, metering, persistence, and idempotency in a single controlled path. Clients can subscribe to real-time channels via Ably but cannot publish directly. Publishing is server-side only.

Architecture: Lean, Auditable, Production-Grade

The SkyeChat codebase clocks in at roughly 9,067 non-blank lines — 7,641 of which are JavaScript or MJS. That's a remarkably small footprint for a platform with this feature surface. Every architectural decision reflects a philosophy of deliberate restraint: fewer moving parts, clean ownership lines, and systems that enforce their own integrity.

The backend's 72 function modules break into four categories: callable endpoints (60), scheduled jobs (2 — an attachment scanner every 10 minutes and a daily retention purge), and internal helper modules handling auth, approvals, rate limiting, and idempotency. The database model follows the same discipline: tenancy, auth/session, chat, attachments, security/ops, enterprise controls, approvals, and billing are each cleanly separated with no cross-concern contamination.

Security Posture

SkyeChat's security model is not a checklist — it is a set of architectural guarantees baked into how the system actually runs.

Server-Side Sessions

Sessions are rows keyed by a hash of cookie + secret. HttpOnly cookies mitigate token theft via XSS. Membership rows are validated on every request — no active row means no access, period.

Approvals Workflow

Sensitive actions return an approval required response. Min approver counts are enforced, requesters cannot approve their own requests, and role gates restrict who can decide on what action type.

Realtime Safety Model

Clients receive Ably capability to subscribe only — never to publish. All writes flow through the controlled messages_send path, enabling DLP, moderation, metering, and idempotency as system guarantees.

Attachment Safety

Dangerous extension blocking, optional external malware scanning, CDR sanitization support, and a scheduled 10-minute scanner loop that classifies and sanitizes attachments even when upstream scanning is slow.

Idempotency & Billing

The messages_send endpoint uses idempotency begin/commit so retries never double-bill or double-log. Usage events feed an immutable ledger plus monthly rollups — dispute-ready by design.

Rate Limiting

DB-backed rate limiting works across serverless cold starts — no in-memory state loss. Widget init includes per-tenant, per-domain, per-IP rate limits. Old rate_limit rows are purged on the daily retention job.

Enterprise Features: SSO, SCIM, DLP, and Approvals

SkyeChat ships four enterprise-tier capabilities that enterprise procurement reviewers specifically look for — and in many cases, their procurement criteria require.

SAML SSO

Full metadata, login, ACS, and SLO endpoints. ACS includes signature and digest algorithm allowlisting — a real enterprise control that prevents weak algorithms from slipping through. Organizations can enforce SSO, blocking email-based authentication for non-allowlisted domains entirely.

SCIM Provisioning

SCIM supports Users and Groups with PATCH and PUT edge handling. Critically, the deprovision path — setting a user's active flag to false — now revokes all server-side sessions immediately. This closes the most common enterprise offboarding gap: the window between HR deprovisioning and system access ending.

DLP Policies

Data loss prevention policies are enforced at the message write level, not as a downstream audit. Because every message passes through the controlled server-side path, DLP cannot be bypassed by client-side logic or widget exploits.

Full Approvals Workflow

Sensitive actions can be gated behind an approval workflow with configurable minimum approvers, four-eyes enforcement (requesters cannot approve their own requests), and role-gated decision authority. The admin console exposes a complete approvals inbox with pending, executed, and rejected states — and every approval-required response path in the UI is handled, with no undefined-key breakage.
