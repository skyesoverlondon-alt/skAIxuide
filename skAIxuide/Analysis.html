WHAT IS
SKAIXU IDE

SkAIxu IDE is a fully engineered, browser-based AI operator environment. Think of it as a command console for AI work — the kind of environment a serious operator, developer, or agency uses to run AI tools, govern model access, switch between specialised applications, and maintain full visibility over what the system is doing. It is not a chatbox. It is not a demo. It is a production-grade intelligent workspace.

The environment hosts a suite of specialised AI tools — including GodKode (the AI coding IDE), Neural-Space-Pro (analysis and research), WebPilePro, SkyeVault, SkyeDocx, and more. Each tool lives inside SkAIxu IDE and inherits its governance, visual system, and gateway configuration. What you acquire when you acquire SkAIxu IDE is the entire operator console, the framework every tool runs inside, and the architecture that governs how AI is accessed and controlled.

You may see the word shell in technical documentation — this is a standard architecture term meaning the host environment that all applications run inside, the same way an operating system is the host environment for its apps. SkAIxu IDE is that host environment. That is a compliment to its engineering, not a diminishment of its scope.

OPERATOR CONSOLE
A full-screen HUD-style AI workspace with 3D neural visuals, glass-panel UI, and a cinematic boot sequence. Feels like a system, not a website.
GATEWAY GOVERNANCE
All AI model access routes through the Kaixu Gateway. One key, one authorisation surface, every tool in the suite controlled from a single configuration point.
TOOL ECOSYSTEM
15+ specialised AI tools accessible from the same navigation — GodKode, Neural-Space-Pro, NexusDB, SkyeVault, WebPilePro, SovereignVariables, and more. One environment, every capability.
OPERATOR TRAINING BUILT IN
Spotlight-based guided onboarding, tour sequences, and opt-in tutorial flows — new operators are trained inside the environment itself. No external documentation required.

Section 02

High-Level Architecture

At the top level, SkAIxu IDE is a browser-native operator console with a dedicated Three.js-ready 3D neural backdrop, a glass-panel foreground layer for all primary UI, a modular app switcher that navigates out to sibling tools, and gateway-aware authentication flows for the Kaixu Virtual Key.

  • A full-screen #canvas-container reserved for the Three.js rendering scene, with pointer-events: none so it never interferes with foreground interaction.
  • A stacked content-layer holding header, modals, and work-surface panels — the HUD effect that makes the system feel like it is floating above the neural background.
  • A pill-based app switcher treating each peer tool as a first-class capability, not a link dump.
  • A Kaixu Gateway-aware authentication surface that centralises the Kaixu Virtual Key and makes that configuration available to every downstream tool via a shared localStorage contract.
  • Thin-server / thick-client architecture: all behaviour encoded in HTML and JavaScript, with Kaixu Gateway and Netlify Functions as the only sanctioned backends.

This architecture is a deliberate engineering choice — one auditable governance surface for all AI traffic, which differentiates a professionally governed AI environment from a collection of standalone demos hitting provider APIs directly.

Section 03

Visual & Interaction Substrate

SkAIxu IDE establishes the visual and interaction substrate that every tool in the suite inherits. Inter for general UI, JetBrains Mono for code surfaces, glassmorphism panels, and carefully tuned motion curves applied consistently across the environment.

  • Glassmorphism panels produce high-contrast, blurred surfaces that read as a professional operator console — not a generic website template.
  • A Three.js neural background lives in the fixed #canvas-container, giving the workspace a living visual substrate while keeping all pointer logic in the foreground.
  • A cinematic splash and boot sequence driven by layered animations (pulse-glow, float-soft, fadeUp, sweep) — the environment boots, it does not just load.
  • A full design token system encoded in CSS variables: colour palette, type scale, spacing, shadow, and glow defined in one place, inherited across every component.
  • Rain canvas, lightning overlay, fog layers, vignette, and film grain — composited in the correct z-index stack so visual effects never block user interaction.

Coordinated HTML, CSS, JavaScript animation, and Three.js integration built to a specific aesthetic standard and maintained across a 15+ tool suite is a fundamentally different category of effort from styled vanilla HTML.

Section 04

Guided Onboarding & Tutorial Engine

SkAIxu IDE ships with a built-in guided onboarding layer implemented as a spotlight-based tutorial system. Not a static help page — live code that manipulates the DOM to walk operators through the environment interactively.

  • #tour-overlay, #tour-spotlight, and #tour-tooltip combine to dim the entire interface except for the specific feature being explained — the same pattern used by enterprise SaaS tools like Intercom and Appcues, implemented natively without a third-party dependency.
  • JavaScript defines sequences of tour steps with coordinates, dimensions, and copy, driven via startTutorial, nextTourStep, and endTutorial functions.
  • A dedicated #tour-offer-modal surfaces an opt-in prompt on first launch with acceptTutorial and skipTutorial handlers — operators are never trapped.
  • The tour infrastructure is reusable: future tools added to the IDE can plug into the same step system without rebuilding the engine.

Operator enablement infrastructure built directly into the product — a reusable training layer for every future tool in the ecosystem, supporting enterprise expectations around discoverability and time-to-competency.

Section 05

Gateway Auth & Governance Surface

Governance is wired into the environment itself. The Kaixu Gateway modal in SkAIxu IDE is the single configuration surface for the Kaixu Virtual Key — the only authorised route to AI capabilities across the entire suite.

  • #auth-modal exposes a Kaixu Gateway configuration panel with branding, a single key input bound to #key-kaixu, and a "Synchronise Gateway" action wired to persistence and handshake logic.
  • The Kaixu key is stored in localStorage as the system-of-record credential and read by downstream tools as KAIXU_VIRTUAL_KEY — one configuration point, every tool.
  • Gateway-only discipline encoded in JavaScript: all AI model traffic routes via Kaixu Gateway and its Netlify Functions, eliminating shadow direct-provider calls.
  • All AI traffic is observable, controllable, and routable through one surface — the codified governance posture that differentiates a professionally managed AI environment from a collection of API playgrounds.
Section 06

Header, App Switcher & Navigation System

The header and app switcher are not ornamental. They encode the mental model of the entire ecosystem and provide the navigational skeleton every tool plugs into.

  • The main header row includes the SkAIxu IDE logomark, undo/redo controls, Original vs Current view-mode toggle, and gateway/tutorial icon controls — all wired into the IDE's internal state.
  • The horizontally scrollable pill-based app switcher links directly to: Neural-Space-Pro, kaixumagic, Nexus-Pro, NexusDBExplorer, NexusConnectHomepage, SkyeVault, SkyeDocx, GodKode, WebPilePro, SovereignVariables, NEWNEW, SkyeBox, SkyeKnife, SmartIDE, and more — each with consistent iconography and styling.
  • The app switcher is not a bookmark bar. It communicates a product: a deliberate ecosystem of specialised AI tools, each with a defined purpose, accessible from one governing environment.
  • By design, SkAIxu IDE is the north star of the suite — it defines shared aesthetics, owns gateway connection, and establishes the mental home for every AI capability in the stack.
Section 07

Integration with GodKode & Peer Tools

GodKode — the flagship AI coding IDE in this ecosystem — is separately valued in its own Analysis document. Within SkAIxu IDE, it is surfaced as the primary coding tool and bound into the environment through shared governance and visual patterns.

  • GodKode is linked from the SkAIxu IDE app switcher, the root portal, ALLAI.html, and Neural-Space-Pro's sidebar — discoverable from every major surface in the ecosystem.
  • Code-level integration uses the shared Kaixu Gateway key in localStorage, a common diagnostics channel (kaixu_events), and matched fonts and glassmorphism so GodKode reads as an extension of the IDE rather than a separate product.
  • Neural-Space-Pro and the remaining peer tools are implemented as autonomous HTML+JS bundles. SkAIxu IDE's navigation and governance patterns bind them into a coherent operator console without requiring architectural coupling.
  • The integration model is a deliberate franchise architecture: one environment, consistent brand discipline, shared governance — independently deployable tools that feel like a single system.
Section 08

Infrastructure & Code Work That Drives Value

Beyond visible features, a category of infrastructure and code work exists inside SkAIxu IDE that substantially increases both its engineering cost and its defensibility as an asset. This work is frequently invisible to non-technical observers but represents real engineering hours — every item below appears with its own hour and £ allocation in the Cost Matrix.

  • Netlify deployment configuration: netlify.toml routing rules, redirect logic, and function configuration — the deployment layer that makes the environment stable and portable at the infrastructure level.
  • Netlify Functions: server-side function layer in netlify/functions/ handling gateway proxying, API key isolation, and server-side model calls — keeping sensitive credentials out of the browser entirely.
  • PWA and service worker implementation: manifest files, service worker registration, and offline-first caching strategies across multiple tools (KaiPrompt, Neural-Space-Pro, projectaegis-skyex) — making the environment installable and resilient.
  • Cross-app localStorage contracts: the shared key schema (KAIXU_VIRTUAL_KEY, kaixu_events, tool-specific state) that allows independent HTML tools to share configuration and context without a backend session layer.
  • Design system implementation: the CSS variable architecture, font stack, animation curves, and component patterns that make 15+ tools look and behave like one product — maintained by convention and propagated from the IDE.
  • Diagnostic and observability layer: events channel, error surface patterns, and health indicators for operator and developer monitoring.
  • Documentation and governance layer: AI-Directives, integration plans, analysis documents, and valuation frameworks — the written infrastructure that allows this codebase to be understood, extended, and transitioned without the original author.

None of this is overhead to be ignored in a valuation. Every item above is engineering hours with a real cost.
